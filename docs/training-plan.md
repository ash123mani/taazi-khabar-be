## Phase 1: Complete the Datasets

### Step 1.1 — Fix quiz setter batch failures

**Problem**: Generating MCQs for 5 articles at once causes HTTP 500. Likely the combined prompt exceeds the teacher model's context window or triggers a server-side timeout.

**Fix**: Reduce batch size from 5 → 2 articles per call. Each call generates 6 MCQs (3 per article × 2). This means 15 batches instead of 6, but each batch is less likely to fail.

Alternatively, generate 1 article per call (3 MCQs). That's 30 calls for 90 records — manageable within the 40 RPM rate limit (~45 seconds total wait time).

**Action**:
```bash
# Edit build_persona_datasets.py: change batch_size from 5 to 2
# Then run quiz setter generation on articles that have summaries but no MCQs yet
python -m app.scripts.build_persona_datasets --quiz
```

## Phase 3: Evaluate Quality

Before deploying to production, we need to know: **is the student model good enough?**

### Step 3.1 — Manual evaluation (quick)

Use the notebook's inference test cell (cell 8). For each adapter:

```python
# Summarizer test
test_prompt = "### Instruction:\nSummarize this UPSC article...\n### Input:\n<real article body>\n### Response:\n"
# Run inference with adapter loaded
# Compare output to teacher model's output for same article
```

Test with 5-10 articles. Compare:
- Does the student produce the right section headers?
- Is the category correct?
- Are key terms relevant?
- Is the syllabus tag reasonable?

Define pass/fail criteria:
- ✅ **Good enough**: Student output is usable, might need minor edits
- ❌ **Not ready**: Missing sections, wrong categories, gibberish

### Step 3.2 — Automated evaluation (if needed)

Script that runs both teacher and student on a held-out test set (10% of dataset) and compares:
- Exact match rate for categorical fields (Category, Syllabus Tag)
- ROUGE-L score for free-text sections (GK Summary, Prelims Focus)
- Classification accuracy for article filter

```python
# Pseudocode
for example in test_set:
    teacher_output = teacher_model(example["input"])
    student_output = student_model(example["input"])
    rouge = calculate_rouge(teacher_output, student_output)
    print(f"ROUGE-L: {rouge}")
```

### Step 3.3 — Decision gate

| Result | Action |
|--------|--------|
| Student matches teacher on ≥80% of test cases | ✅ Deploy to production |
| Student matches on 50-80% | ⚠️ Deploy alongside teacher (A/B test), monitor |
| Student matches on <50% | ❌ Need more data or different base model |

## Phase 4: Deploy to Production

### Step 4.1 — Upload adapter to NVIDIA NIM

NVIDIA NIM supports custom LoRA adapters via their NVCF (NVIDIA Cloud Functions) platform. The process:

1. Package adapter files (adapter_config.json + adapter_model.safetensors) into a tar.gz
2. Upload via NVIDIA NIM console or API
3. Note the adapter name/ID returned

### Step 4.2 — Configure orchestrator to use LoRA

In `app/ai/providers/nim.py`, `complete_with_lora()` is already implemented. We need to update `orchestrator.py` to call it instead of `complete()`.

```python
# orchestrator.py (after deployment)
async def summarize_article(self, article_body: str, ...):
    model_config = registry.get_active_model("summarizer")
    api_key, base_url = settings.get_persona_credentials("summarizer")
    
    system, prompt = summarizer.build_prompt(article_body)
    
    # OLD: response = await self._provider.complete(...)
    # NEW:
    response = await self._provider.complete_with_lora(
        prompt=prompt,
        system=system,
        lora_adapter="taazi-summarizer-v1",  # adapter name from NVIDIA
        api_key=api_key,
        base_url=base_url,
    )
    
    parsed = summarizer.parse_response(response.text)
    return parsed
```

### Step 4.3 — Gradual roll-out

| Day | Action | Risk |
|-----|--------|------|
| 1 | Switch article filter to LoRA (lowest risk — YES/NO only) | Low |
| 2 | Monitor: compare filter decisions vs old model | Low |
| 3 | If filter OK, switch summarizer to LoRA | Medium |
| 5 | Monitor summary quality. Rollback if issues | Medium |
| 7 | Switch quiz setter to LoRA | High |
| 10 | Full LoRA deployment | - |

### Step 4.4 — Rollback plan

If quality drops, switch back to teacher model:
```python
# In orchestrator.py, use a config flag:
USE_LORA = os.getenv("USE_LORA", "false").lower() == "true"
if USE_LORA:
    response = await self._provider.complete_with_lora(...)
else:
    response = await self._provider.complete(...)
```

Setting `USE_LORA=false` in `.env` instantly reverts to the original cloud model.
